Setup Guide

Getting started

Follow the steps below to run the Discord bot locally or via Docker. Supabase memory is optional — the bot logs a warning and continues without it.

Prerequisites

What you need

🐍
Python 3.11+

Required for zoneinfo (stdlib) and type syntax used throughout the project.

🤖
LLM Backend

Either a Gemini API key from Google AI Studio, or a locally running Ollama instance. See the model notes below before choosing.

📰
GNews API key

Free tier at gnews.io. Note: the free plan is localhost-only — you'll need a paid plan to run in Docker or on a server.

💬
Discord Bot

A bot application in the Discord Developer Portal with Message Content Intent enabled. See the Discord section below.

📅
Google Calendar

A GCP service account with Viewer access to your calendar. The JSON key is Base64-encoded into an env var. See the Calendar section below.

🗄️
Supabase (optional)

Enables long-term semantic memory. Without it, the agent still works — it just won't recall past briefings across restarts.

Installation

Step-by-step setup

1

Clone the repository

git clone https://github.com/matthewshan/adk-playground.git
cd adk-playground
2

Install Python dependencies

python3 -m pip install --user -r requirements.txt
3

Configure environment variables

Copy the example env file and fill in your secrets. Never commit .env.

cp daily_briefing/.env.example daily_briefing/.env
# then edit daily_briefing/.env with your values

See the environment variables table and per-service setup sections below for what each value should be.

4

Run smoke tests (optional but recommended)

These call APIs directly without running the full agent. Missing API keys are skipped automatically — you'll see which ones pass.

# All tool APIs
python3 daily_briefing/smoke_tests/test_apis.py

# Sports tool specifically (no API key required)
python3 daily_briefing/smoke_tests/test_sports.py

# Full agent run — calls the LLM and all tools, prints digest to stdout
python3 daily_briefing/smoke_tests/test_agent.py
5

Start the Discord bot

The bot posts the daily briefing at 7 AM ET and responds to direct messages in the configured channel.

python3 -m daily_briefing.discord_bot

On startup you should see a log line like Logged in as YourBot#1234. If you see a token error, double-check DISCORD_BOT_TOKEN in your .env.

CLI debug mode (no Discord needed)

To print a briefing to stdout without a bot token, run python3 -m daily_briefing.main. This is the fastest way to verify the agent and tools are working.

Configuration

Environment variables

All secrets live in daily_briefing/.env. Copy from daily_briefing/.env.example.

Variable Required Description
BACKEND optional gemini (default) or ollama. Selects the LLM backend.
GEMINI_API_KEY if Gemini Google AI Studio API key. Get one at aistudio.google.com/app/apikey.
GEMINI_MODEL optional Defaults to gemini-3.1-flash-lite. Recommended: gemini-2.5-flash-lite (1 000 RPD free tier). Do not use gemma-4-* models — they do not support ADK tool/function calling.
OLLAMA_API_BASE if Ollama Ollama server URL, e.g. http://127.0.0.1:11434.
OLLAMA_MODEL if Ollama Model tag. Recommended: qwen2.5:7b (~4.5 GB, excellent tool calling). Do not use qwen3:8b — its thinking/reasoning mode produces extremely verbose output.
GNEWS_API_KEY yes GNews API key from gnews.io. Free tier: 100 requests/day, localhost only. A paid plan is required to run from Docker or a server.
DISCORD_BOT_TOKEN yes Discord bot token. Discord Developer Portal → Your App → Bot → Reset Token.
DISCORD_BOT_CHANNEL_ID yes ID of the Discord channel the bot listens in and posts to. Right-click the channel → Copy Channel ID (requires Developer Mode in Discord settings).
GOOGLE_CALENDAR_ID yes The calendar's email address, e.g. yourname@gmail.com or a calendar-specific address found in Google Calendar → Settings → Calendar ID.
GOOGLE_SERVICE_ACCOUNT_JSON_BASE64 yes Base64-encoded service account JSON key file. Encode with: base64 -w 0 service-account.json. See the Google Calendar section below for how to obtain the JSON.
SUPABASE_URL optional Supabase project URL (e.g. https://<ref>.supabase.co). Enables long-term memory. Bot degrades gracefully if absent.
SUPABASE_SERVICE_ROLE_KEY optional Supabase service_role key (not the anon key). Has full DB access and bypasses row-level security — keep it in .env only, never commit it.
Discord

Discord bot setup

1

Create a Discord Application

Go to discord.com/developers/applications, click New Application, and give it a name.

2

Create a Bot and copy the token

In your application, go to Bot in the left sidebar. Click Reset Token, confirm, and copy the token. This is your DISCORD_BOT_TOKEN. You only see the token once — save it immediately.

3

Enable Message Content Intent

Still on the Bot page, scroll down to Privileged Gateway Intents and toggle on Message Content Intent. Without this, the bot can't read what users type.

4

Invite the bot to your server

Go to OAuth2 → URL Generator. Under Scopes, select bot. Under Bot Permissions, select: Send Messages and Read Message History. Copy the generated URL and open it in a browser to add the bot to your server.

5

Copy the channel ID

In Discord settings, go to Advanced and enable Developer Mode. Then right-click the channel you want the bot to use → Copy Channel ID. This is your DISCORD_BOT_CHANNEL_ID.

Google Calendar

Google Calendar service account setup

The agent accesses your calendar via a GCP service account — no OAuth login required at runtime. The full Terraform-based provisioning is documented in cloud-infrastructure/terraform-adk-agents. The manual steps are below.

1

Create a GCP project and enable the Calendar API

Go to the Google Cloud Console, create a new project, then navigate to APIs & Services → Library and enable the Google Calendar API. A billing account is required to create a project, but the Calendar API is free.

2

Create a service account

Go to IAM & Admin → Service Accounts → Create Service Account. Give it a name (e.g. daily-briefing-agent). No project-level IAM role is needed — calendar access is granted by sharing the calendar directly.

3

Create and download a JSON key

Click the service account → Keys tab → Add Key → Create new key → JSON. Save the downloaded file. Then Base64-encode it:

base64 -w 0 service-account.json

Paste the output as GOOGLE_SERVICE_ACCOUNT_JSON_BASE64 in your .env.

4

Share your calendar with the service account

Open Google Calendar, click the three dots next to the calendar you want to share → Settings and sharing → Share with specific people. Add the service account email (looks like daily-briefing-agent@your-project.iam.gserviceaccount.com) with See all event details permission.

5

Set the Calendar ID in .env

Find your calendar's ID in Google Calendar → click the calendar → Settings → Calendar ID. For your primary calendar this is usually just your Gmail address. Set it as GOOGLE_CALENDAR_ID.

Docker

Running with Docker

Discord Bot (primary — includes scheduling)

docker build -t daily-briefing-bot -f daily_briefing/Dockerfile.bot .
docker run --env-file daily_briefing/.env daily-briefing-bot

ADK Web UI (dev / testing)

docker build -t daily-briefing-dev -f daily_briefing/Dockerfile.dev .
docker run --env-file daily_briefing/.env -p 8000:8000 daily-briefing-dev
# Then open http://localhost:8000 and select "daily_briefing"
Running the ADK Web UI without Docker

Run from the repo root — ADK discovers daily_briefing/ automatically:
adk web --host 0.0.0.0 --port 8000

GNews free tier + Docker

The GNews free plan only permits requests from localhost. If you run the bot in Docker or on a server you'll need a paid GNews plan, or the news section of the briefing will fail.

Optional

Setting up Supabase memory

Long-term memory is optional. Without it, the agent works fine but won't recall past briefings. Full details in docs/integrations/supabase-vector-adk.md.

1

Create a Supabase project

Sign up at supabase.com and create a new project. Note the Project URL and the service_role key (not the anon key) from Settings → API → Project API keys.

Use the service_role key, not anon

The agent runs server-side and needs cross-session read/write access. The service_role key bypasses row-level security — keep it in .env only, never commit it.

2

Run the one-time schema setup

In the Supabase SQL Editor, run the following. This creates the memory table, an HNSW index, and the similarity-search RPC function the Python client calls.

-- Enable pgvector (must run first — not auto-enabled)
CREATE EXTENSION IF NOT EXISTS vector;

-- Drop old objects if re-running after a dimension change
DROP FUNCTION IF EXISTS match_agent_memory(vector, int);
DROP TABLE IF EXISTS agent_memory;

-- Memory table
-- Embeddings use 1536 dims (gemini-embedding-001 truncated from 3072
-- to stay within pgvector's HNSW/IVFFlat index limit of 2000)
CREATE TABLE agent_memory (
  id          bigserial    PRIMARY KEY,
  session_id  text         NOT NULL,
  content     text         NOT NULL,
  embedding   vector(1536),
  created_at  timestamptz  DEFAULT now()
);

-- HNSW index for approximate nearest-neighbour search
-- (handles dynamic inserts correctly, unlike IVFFlat)
CREATE INDEX ON agent_memory USING hnsw (embedding vector_cosine_ops);

-- RPC function used by the Python memory tool for similarity search
CREATE OR REPLACE FUNCTION match_agent_memory(
  query_embedding vector(1536),
  match_count     int DEFAULT 5
)
RETURNS TABLE (id bigint, session_id text, content text, similarity float)
LANGUAGE sql STABLE
AS $$
  SELECT   id, session_id, content,
           1 - (embedding <=> query_embedding) AS similarity
  FROM     agent_memory
  ORDER BY embedding <=> query_embedding
  LIMIT    match_count;
$$;
3

Add Supabase keys to .env

SUPABASE_URL=https://your-project-ref.supabase.co
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
4

Verify with the memory smoke test

python3 daily_briefing/smoke_tests/test_memory.py
