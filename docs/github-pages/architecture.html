    Scheduler isolation

    The scheduler runs under its own "scheduler" user ID, completely isolated from per-user conversation sessions. The scheduler session resets each morning so the agent starts fresh and doesn't accumulate a growing context window over time.

    Codebase

    Module layout

    The project follows a strict separation: raw HTTP clients in apis/, ADK-registered functions in tools/.

    Module Role Key contents
    agent.py Agent wiring make_agent(), root_agent, model selection (Gemini/Ollama), after_agent_callback
    instruction.md System prompt The only place agent behavior lives. Editable without code changes. Defines briefing format, conversational rules, and live score detection.
    discord_bot.py Bot entrypoint Scheduled task (7 AM ET via discord.ext.tasks), on_message handler, session and lock management, message chunking
    tools/ ADK tool functions weather.py, news.py, sports.py, calendar_events.py — plain Python callables registered with the ADK Agent
    apis/ Raw HTTP clients open_meteo.py, gnews.py, espn.py, thesportsdb.py, google_calendar.py, supabase.py
    memory/ Memory service supabase_memory_service.py — implements ADK BaseMemoryService with pgvector insert + similarity search
    smoke_tests/ Integration tests Runnable scripts: test_agent.py, test_apis.py, test_sports.py, test_discord_bot.py, test_memory.py
    Design

    Key decisions

    Click each card to read the reasoning behind architectural choices.

    The Discord bot handles both the 7 AM daily briefing (via discord.ext.tasks) and real-time conversation. A single long-running Kubernetes Deployment replaces the original CronJob + webserver design. This eliminates cold-start latency, simplifies state sharing, and means one image to build and maintain.

    The ADK agent's only job is to produce a text response. discord_bot.py is solely responsible for chunking at 2 000 characters (Discord's limit) and calling channel.send(). This keeps the agent portable and testable via CLI (python -m daily_briefing.main) without needing a Discord connection.

    ADK tools are just regular Python functions — no @tool decorator is needed. This avoids a critical pitfall: complex type annotations (dataclasses, unions with None, nested generics) generate $defs/$ref/anyOf JSON schemas that Gemini silently rejects during function calling. The rule: use only str, int, float, bool in tool signatures.

    ESPN's unofficial API covers NFL (Detroit Lions) and MLB (Toronto Blue Jays) well, but CFL (Hamilton Tiger-Cats) coverage is incomplete. TheSportsDB is used as a targeted fallback specifically for CFL data. Off-season detection is built into the sports tool to avoid producing confusing output when leagues aren't active.

    instruction.md is the single source of truth for agent behavior. It's loaded at startup with Path(__file__).parent / "instruction.md". Prompt tweaks don't require code changes or a Docker rebuild — edit the file and restart the bot. This also makes the prompt easy to diff in version control and review in PRs.

    Setting BACKEND=ollama swaps in a local model via LiteLlm. This was added to work around Gemini free-tier rate limits during development. Recommended Ollama model: qwen2.5:7b (~4.5 GB, excellent tool-calling support). Avoid llama3.1:8b (poor tool use) and Gemma models (don't support ADK function calling at all).
